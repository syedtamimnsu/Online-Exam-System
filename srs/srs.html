1. Introduction


1.1 Purpose

The document describes the software requirement specifications of the tentatively name project - “Online Reading Helper”. The primary purpose of the system is to create an online reading helper where teachers can create student-cohorts and assign them reading materials in the form of pdfs. The teachers will be able to gather personalized data about the reading ability of the students on the content provided by the teacher. Thus, they will be able to give more targeted help to the students regarding the reading materials.
The data will be gathered by the system by assessing the student's attention towards the content while he is reading it. The teachers will also be able to assign quizzes to the reading materials to gather more data as to whether he actually understood the content or not.

1.2 Document Conventions

The font used in the printed document is Arial of size 11. The web view document will have the default font setting of bootstrap. The text is made bold and of bigger size for highlighting different sections of the document. In this document, every functionality is equally important and every requirement has its own priority.

1.3 Intended Audience and Reading Suggestions

The document is intended towards the developers, testers, designers who are making the system and users who are using it. This document will serve as a reference document for every team who are partaking in making the project. They will coordinate every activity that take place in the software engineering process of the system and will be guided by Dr. Nabeel Mohammed.

1.4 Product Scope

The system will be able to help any teachers and learners on gaining more value from reading materials. It will solve the recent problem of students not gaining as much value or knowledge from books or reading materials as they gain from interactive learning like through videos. The teachers will be able to assign short chunks of reading materials to student-cohorts and also assign quizzes to them through our system. The system will generate personalized data about the students on their reading ability. Thus, teachers will be able to provide more targeted help towards students.

Bootstrap

2. Overall Description


2.1 Product Perspective

Online reading helper is a project that aims to make reading content more viable to students. It aims to make sure readers read and understand the contents assigned to them effectively. Learners today learn more efficiently with interactive systems like videos. But, for any new thing that comes out in the market it is always documented first and filmed latter. So, to keep up with the world it is necessary to improve our reading ability.
Our project aims to achieve such goal by making sure that learners pay attention to the content when they are reading it and they understand it through performing in quizzes. Teachers will be able to monitor the performance of the students and give them more targeted guidance. Our project is self-contained and is not a part of any other bigger systems in the market.

2.2 Product Functions

The major functions of the system are provided below:

  • Users can sign up and login through google.
  • Users logged in as teachers can create a student-cohort and upload reading materials to the cohort with the option to attach quiz assessments to it.
  • Teachers will be able to add or remove students in a cohort. They can also add other previously created cohorts to a new cohort.
  • Users logged in as students can read the materials in app and sit for quizzes after they are done with their reading.
    •
2.3 User Classes and Characteristics

There will be three different types of actors for our system. They are: users, admins, and the system itself.
There will generally be two different types of users for our system: teachers, learners. Users logged in as teachers can create cohorts and upload reading materials and take quizzes on them. They will be able to view their students' performances and identify where they are struggling. Users logged in as learners will be able to read the materials provided to them by the teachers and sit for quizzes.
System admins will hold the highest power and they will be able to delete accounts. The system itself will be able to perform some tasks like, collect attention data of students, recognize students faces, send learners' performance data to teachers.

2.4 Operating Environment
  • Platform supported: Web and Android
  • Database: SQLite
    •
2.5 Design and Implementation Constraints

The system will have a web app and android app, but no IOS app. Users can only get notifications form the android app, for example, new reading material published, approaching deadline, etc. Users cannot use other features inside the mobile app.

2.6 User Documentation

The product will come with a user manual which will explain how to use the system such as how to create a student-cohort, add students, create an assessment, etc.

2.7 Assumptions and Dependencies

Users must have a stable and working internet connection. Web users must have a working browser to use the product. Mobile users must have android version 6+ installed.



3. External Interface Requirements


3.1 User Interfaces

The user will start at the login page, where they will find a button for google login. Users can log in using a valid google account. After they are logged in, they can create a student cohort where they can add other users(students). Inside the cohort, the user (Teacher) can upload reading materials and add questions for assessment. They(teacher) can also check students attentiveness fo each section of the uploaded reading material, quiz marksheet, and provide targeted help to the students.
Learners will also start in the login page, where they can login through google. After login user can see his/her profile. They can see the list of cohorts that they are a part of, their pending reading materials and assesments. They will also be able to see their performances under each cohort.

3.2 Hardware Interfaces

Our system will not have any hardware components so, it does not have any hardware interface. The hardware connection between the database server and application is managed by the underlying operating system of the internet device.

3.3 Software Interfaces

The system is self-contained and no data is supposed to share with any third party. The communication of the mobile application with the database consist of only reading the data and notifying students of pending tasks, no modification is allowed. The communication between the web application and database consist of both reading and modifying.

3.4 Communications Interfaces

The communication between the different parts of the system is important since they depend on each other. However, in what way the communication is achieved is not important for the system and is therefore handled by the underlying operating system for the mobile application.



4. System Features


4.1 Log in using Google

Use Case: Login using Google
Actor(s): User
Purpose: The user can login to this system using their Google account, and then use the system.
Type: Primary

Typical course of events:
Actor Action System Response
1. User clicks on the login button from login page. 2. System redirects to the google login page
3. User chooses a google account 4. System will response by verifying the credentials through google API.
Alternative Course of Events

Section 3: If the user enter incorrect google credentials, the system will redirect to login page.

4.2 Create a student-cohort

Use Case: Create a student-cohort
Actor(s): Teacher
Purpose: After logging into the system, the user can create a student cohort by clicking the "create student-cohort" button
Type: Primary

Typical course of events:
Actor Action System Response
1. User clicks “Create a student-cohort” button. 2. The system provides the create a student-cohort form.
3. User fills up the necessary information and clicks the submit button. 4. System will show a successful message and the cohort will be added to the dashboard.
Alternative Course of Events

Section 2: The user login session might be expired, the system will redirect to the login page

Section 4: The system will show an error if the form is not filled properly/if some information is missing.

4.3 Upload reading materials to a cohort

Use Case: Upload reading materials to a cohort
Actor(s): Teacher
Purpose: The teacher of a student-cohort can upload reading materials for that cohort.
Type: Primary

Typical course of events:
Actor Action System Response
1. Teacher clicks on the "Upload PDF" button under a student-cohort. 2. The system pops up the upload file window.
3. Teacher drags or select the file to be uploaded. 4. System will show a successful message and will upload the PDF to the cohort's dashboard.
Alternative Course of Events

Section 4: The system will show an error if the teacher uploads a file that is not PDF.

4.4 Add assesments to the reading material

Use Case: Add assesments to the reading material
Actor(s): Teacher
Purpose: The teacher of a student-cohort will have the option to add assesments to the uploaded reading material.
Type: Primary

Typical course of events:
Actor Action System Response
1. Teacher clicks on the "Add an assessment" button after uploading a PDF. 2. The system redirects to the assessment creation form.
3. Teacher will fill up the form and click submit. 4. System will show a successful message and will add that assessment to that cohort's dashboard under the reading material.
Alternative Course of Events

Section 4: The system will show an error if the form is not filled properly or if some information is missing.

4.5 Add users (students) to the cohort

Use Case: Add users (students) to the cohort
Actor(s): Teacher
Purpose: The Teacher can add students to the student-cohort.
Type: Primary

Typical course of events:
Actor Action System Response
1. Teacher clicks the "add student" button. 2. System redirects to the add student page.
3. Teacher will enters the student's email address and then click submit button. 4. System will add the student to that cohort.
Alternative Course of Events

Section 4: There can be an error because the entered email might be invalid or unknown.

4.6 Edit assessments

Use Case: Edit assessments
Actor(s): Teacher
Purpose: The teacher can edit a created assessment.
Type: Primary

Typical course of events:
Actor Action System Response
1. Teacher clicks “Edit assessment”. 2. The system provides the assessment page, which is editable
3. User does necessary changes to the assessment and saves it. 4. System will correct update the database and show the corrected assessment.
Alternative Course of Events

Section 4: The system will show an error if there were some invalid changes

4.7 Add MCQs to the assessment

Use Case: Add MCQs to the assessment
Actor(s): Teacher
Purpose: The teacher of a student-cohort can add MCQ questions in the assesment and specify the correct answer.
Type: Primary

Typical course of events:
Actor Action System Response
1. Teacher clicks "Add MCQs" 2. The system will provide a text box to write the questions, answer options, and assign the mark.
3. Teacher will write the question and specify the marks as well as the correct answer for that question and click save. 4. System will save that question and add it to the assessment.
Alternative Course of Events

Section 3: If the specified mark for an MCQ is more than the total mark of the assessment, then an error will be shown.

Section 4: If the teacher clicks the save button without specifying the correct answer and mark, then it will show an error.

4.8 Add voiced-answers to the assessment

Use Case: Add voiced-answers to the assessment
Actor(s): Teacher
Purpose: The teacher of a student cohort can add voiced-answers for the questions and specify the possible correct answers by text.
Type: Primary

Typical course of events:
Actor Action System Response
1. Teacher clicks "Add voiced-answers" 2. The system will provide a form for writing the question and specifying its answer by text.
3. Teacher will write the question and type in the possible correct answer. 4. System will save that question and add it to the assessment.
Alternative Course of Events

Section 3: If the specified mark for voiced-answer question is more than the total mark of the assessment, then an error will be shown.

Section 4: If the teacher clicks the save button without specifying the correct answer and mark, then it will show an error.

4.9 View cohort performance

Use Case: 4.8 View cohort performance
Actor(s): Teacher
Purpose: The teacher of an exam cohort can view the assesment performance of the students. The window will have the breakdowns for all marks and detailed statistical percentages.
Type: Primary

Typical course of events:
Actor Action System Response
1. Teacher clicks "View cohort performance" 2. The system will redirect to a performance statistics window
4.10 Read uploaded reading materials

Use Case: Read uploaded reading materials
Actor(s): Student
Purpose: The student of a student-cohort can start reading the materials in app provided that it is still within the specified deadline set by the teachers.
Type: Primary

Typical course of events:
Actor Action System Response
1. Student clicks on the reading material that he wants to read. 2. The system pops up reading window.
Alternative Course of Events

Section 2: The system will show an error if the reading material is not available to the database anymore.

4.11 Take the assesment

Use Case: Take the assesment
Actor(s): Students
Purpose: The students of a student-cohort can take the assesment if they are done reading the materials.
Type: Primary

Typical course of events:
Actor Action System Response
1. Student clicks "Start the assesment" 2. The system will redirect the student to the exam window.
3. The student will answer the mcqs by ticking the correct option, and for voiced-answers, they will answer using their microphone and record their answer in app. 5. System will save the answers.
4. The student will click the submit button after the exam finishes. 6. System will also save the answers and return the marks.
Alternative Course of Events

Section 5: If internet fails during the exam, the assesment might be paused.

4.12 Log out

Use Case: Log out
Actor(s): Users
Purpose: The users can log out of their account using the logout button.
Type: Primary

Typical course of events:
Actor Action System Response
1. User clicks the Logout button. 2. The system will redirect the user to the login page.


5. Other Nonfunctional Requirements


5.1 Performance Requirements
  • Both android app and web app should use the same backend to ensure good performance and synchronization.
  • The system should be well optimized to tackle large volume of traffic at once.
  • System should be available 24/7.
5.2 Safety Requirements
  • Errors should be minimized and an appropriate error message that guides the user from an error should be provided.
  • Validation of users input must be highly maintained.
5.3 Security Requirements
  • The system must provide a high level of security and integrity of the data.
  • Only users with active Google accounts should be permitted to establish accounts and view their user pages, and only authorized workers such as admins should granted access to confidential data.
5.4 Software Quality Attributes
  • Availability, Reliability, and Usability are the three main qualities of high-quality software.
  • As the system is anticipated to operate 24/7, High accessibility is crucial.
  • A simple and interactive user interface is developed to make navigation easy.
  • The error notice that is presented is more detailed and simpler to comprehend.
5.5 Business Rules
  • The system will be available publicly.
  • Anyone can use the system free of cost.
